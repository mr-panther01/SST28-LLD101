❌ The Problem: Inheritance Gone Wrong
We start with a base class:
class NotificationSender {
    void send(Notification n);
}


Then we have three subclasses:
EmailSender       SmsSender       WhatsAppSender
   ↓                 ↓                 ↓
- Truncates      - Ignores         - Throws error if
  body             subject            phone not in + format


Now imagine you’re a caller using NotificationSender. You expect:
- The message body to be preserved.
- The subject to be used.
- The phone number to be accepted.
But surprise!
- EmailSender silently chops off your message.
- SmsSender throws away the subject.
- WhatsAppSender crashes if the number isn’t in E.164 format.
So what’s the issue?
👉 Subtypes break the promise of the base class.
You can’t substitute one sender for another without knowing its quirks. That’s a violation of LSP.

✅ The Solution: Interface + Encapsulated Behavior
Instead of forcing all senders into one inheritance tree, we define a clean interface:
interface NotificationSender {
    void send(Notification n);
}


Each sender now implements this interface and handles its quirks internally:
EmailSender       SmsSender       WhatsAppSender
   ↓                 ↓                 ↓
- Truncates      - Ignores         - Logs error if
  visibly          subject            phone invalid
- No surprises   - Consistent      - No crash


Now the caller can use any sender without worrying about runtime surprises. That’s LSP in action.

            [NotificationSender]  ← abstract base class
                  send(Notification)

        ┌────────────┬──────────────┬─────────────┐
        ↓            ↓              ↓
[EmailSender]   [SmsSender]   [WhatsAppSender]
  - silently      - ignores      - throws error if
    truncates       subject        phone not in +E.164
    body            field          format

Caller expects:
- subject used
- body preserved
- phone accepted

But:
- Email changes meaning
- SMS discards subject
- WA tightens precondition
→ Caller must know subtype quirks
→ Violates substitutability

            [NotificationSender] ← interface
                  send(Notification)

        ┌────────────┬──────────────┬─────────────┐
        ↓            ↓              ↓
[EmailSender]   [SmsSender]   [WhatsAppSender]
  - truncates     - ignores      - logs error if
    visibly         subject        phone invalid
  - preserves     - consistent   - no exception thrown
    contract        contract       → safe fallback

Caller expects:
- Same Notification structure
- No runtime surprises
- Consistent behavior

→ All senders uphold base contract
→ Caller can substitute freely
→ LSP preserved
